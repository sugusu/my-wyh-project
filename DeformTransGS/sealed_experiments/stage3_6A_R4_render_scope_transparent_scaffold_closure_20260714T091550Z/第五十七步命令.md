# 二、官方源码恰好支持这个风险

官方 `render_dataset_mask()` 对不参与场景 mask 的对象会同时设置：

```python
obj.hide_viewport = True
obj.hide_render = True
```

参与 mask 的对象则两个都设为 `False`。 ([GitHub][1])

但是 `export_merged_mesh()` 只修改：

```python
obj.hide_viewport
```

随后调用：

```python
bpy.ops.wm.obj_export(
    export_selected_objects=False,
    ...
)
```

它没有同步设置 `hide_render`。 ([GitHub][1])

Blender 的 OBJ 导出文档说明，在未启用“仅选中对象”时，导出目标是场景对象，而不是根据官方 Cycles/EEVEE render-mask 的 `hide_render` 语义自动复现渲染对象集合。 ([Blender 文档][2])

所以存在一个非常合理的风险：

[
\boxed{
\text{OBJ exported blocks}
\neq
\text{mask-render visible blocks}
}
]

注意，我说的是**存在强风险**。

R2 已经给了实际证据：

```text
BLOCK_SCOPE_MIXED = YES
```

例如有 block：

[
median\ mask\ support=1
]

也有 block：

[
median=0
]

# 三、这会怎样杀死 R3？

```text

    ↓

    ↓
background
```

```text

    ↓

    ↓
```

```text

```

于是 transparent pixel 上记录：

```text

```

* glass block precision = 0；
* glass component precision = 0；
* glass face precision = 0。

**即使透明 mesh 真实存在于 OBJ 中。**

所以：

```text

```

在当前 protocol 下仍然不能证明：

> transparent geometry 不存在。

它只能证明：

> **在“所有 OBJ block 都作为遮挡体”的 z-buffer 中，没有 visible face label 可以恢复 transparent mask。**

# 我的最终评价

我们现在确实已经在 GT mesh 基础设施上花了很多轮。

所以我同意设一个**硬终点**：

> **R4 是最后一轮。**

理由是 R3 仍然存在一个明确、可证伪的遮挡范围漏洞，而官方代码确实对 mask render 同时控制 `hide_viewport` 和 `hide_render`，mesh export 的对象范围处理并不等同于该 render visibility 逻辑。 ([GitHub][1])

因此再做一次“先 scene render scope、再 transparent scope”是合理的。

但是：

* **RS1 FAIL：GT mesh 路线永久停止。**
* **RS1 PASS、RS2 FAIL：GT mesh 路线永久停止。**
* **RS1/RS2 PASS：不再审计，直接 KIOT vs opacity-linear 生死门。**

还有最重要的一句：

> **当前 KIOT 仍然是 UNDECIDED，但它已经只能被称为 controlled-carrier-supported。**

如果 R4 失败，我不建议继续找第三种 scaffold 救 KIOT。那时应该正式回到课题层面，重新评估这条创新主线是否值得继续。
