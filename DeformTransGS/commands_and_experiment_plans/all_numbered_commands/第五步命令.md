# 二、真正的问题是：所有属性梯度同时为 0

现在：

```text
O grad = 0/0/0
C grad = 0/0/0
V grad = 0/0/0
```

这非常关键。

如果只有 V=0，我会怀疑 local-view formula（局部视角公式）。

如果只有 O=0，我会怀疑 opacity saturation（不透明度饱和）。

但：

[
O=C=V=0
]

同时发生。

我的第一判断是：

> **共享渲染图或 loss graph（损失计算图）发生了 detach。**

因为 TSGS 官方 renderer 本身直接取得 `pc.get_opacity`，通过 PyTorch 运算计算 `colors_precomp`，然后把 opacity 和预计算颜色直接传给 FirstSurface rasterizer。 ([GitHub][1])

更重要的是，TSGS 使用的 `diff-first-surface-rasterization` 明确实现为 `torch.autograd.Function`，CUDA backward 会计算并返回 `grad_colors_precomp` 和 `grad_opacities`。标准 3DGS rasterizer 的官方实现同样明确返回预计算颜色和 opacity 的梯度。 ([GitHub][2])

因此从官方实现推断：

> **“rasterizer 天生不给 O/C 梯度”不是首要怀疑。**

更可能是你新写的 `real_oracle/render_adapter.py` 或 loss 路径中出现了：

```python
.detach()
```

或者：

```python
torch.no_grad()
```

或者：

```python
tensor.cpu().numpy()
```

然后重新：

```python
torch.tensor(array)
```

或者：

```python
torch.tensor(existing_tensor)
```

或者把属性写入一个不带梯度的中间 buffer：

```python
buffer[:, ...] = opacity
```

再使用 buffer。

还有一种很危险的情况：

```python
render(...)
→ save npy
→ load npy
→ compute loss
```

这样 forward 图肯定彻底断掉。

## 我现在的真实判断

**这条主线目前还没到该杀的时候。**

原因非常简单：

我们现在终于第一次拥有一个可信 GT：

[
\boxed{\tau\ error\approx5.7\times10^{-8}}
]

而官方 TSGS 渲染路径确实把 opacity 和预计算颜色送进可微 rasterizer；其 CUDA autograd 包装明确返回 opacity 和 color 的梯度。 ([GitHub][1])

所以现在的：

```text
O=0
C=0
V=0
```

**更像一个共享代码图断裂，而不是“Gaussian 属性无法优化”。这是基于三类属性同时失效以及官方 backward 接口做出的推断。**

下一轮我最关注的其实只有 5 行：

```text
O first broken edge
C first broken edge
V first broken edge
direct rasterizer opacity grad
direct rasterizer color grad
```

只要 direct rasterizer 的 O/C gradient 非零，基本就能把锅锁死在我们自己的 `real_oracle` adapter/loss 上。

**先修真实实验管线。科学结果现在仍然是 UNTESTED。**
