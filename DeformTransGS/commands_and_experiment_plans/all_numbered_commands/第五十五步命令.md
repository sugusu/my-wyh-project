## 一、按官方源码，理论上不应该存在一个未知 Sim(3) 对齐

TransLab 官方生成代码直接把 Blender 场景 mesh 导出为 `scene_mesh.obj`。保存 COLMAP 相机时，它构造 Blender camera-to-world，然后只通过 `blender2opencv=diag(1,-1,-1,1)` 改变**相机局部坐标轴约定**，再求逆得到 world-to-camera 的 (R,T)。它没有对世界中的 mesh 顶点应用额外平移、旋转或尺度变换。([GitHub][1])

COLMAP 官方定义同样明确：`images.txt` 中的 quaternion 和 translation 表示 **world→camera**，相机坐标为 X 向右、Y 向下、Z 向前。([COLMAP][2])

TSGS 读取 COLMAP 时直接使用 qvec/tvec，仅将旋转矩阵转置后存成 Graphdeco/CUDA 约定。([GitHub][3])

所以从源码链推断：

[
\boxed{
\text{scene_mesh Blender world}
===============================

# \text{COLMAP world}

\text{TSGS world}
}
]

至少**不应该凭空存在未知 Sim(3)**。

因此：

[
IoU=0.2575
]

更应该首先怀疑：

1. mask-camera 文件匹配；
2. triangle silhouette rasterizer（网格轮廓光栅器）的投影矩阵或行列向量约定；
3. 图像 Y 轴、像素中心、裁剪语义；
4. OBJ 导出对象范围和 Blender mask 渲染对象范围不完全一致。

而不是继续猜坐标变换。

## 二、官方源码里确实还有一个值得警惕的 scope asymmetry（范围不对称）

`render_dataset_mask()` 对不参与 mask 的对象同时设置：

```text
hide_viewport = True
hide_render = True
```

而 `export_merged_mesh()` 主要修改的是：

```text
hide_viewport
```

随后：

```text
obj_export(export_selected_objects=False)
```

导出整个场景 mesh。([GitHub][1])

因此我做一个**源码支持的推断**：

> mesh export 与 mask render 的对象可见性控制代码并非完全相同。

这不证明 `scene_mesh.obj` 一定包含额外隐藏 mesh，但必须在本地数据上查。

你的：

```text

816 connected components
```

说明这个风险完全值得查。

而且我建议彻底绕过“先渲染 silhouette 再猜原因”的方式。

用一个非常简单但很强的事实：

> 对于完整场景 union mask，任意位于相机前方、投影进图像的场景 mesh 表面点，其投影位置理论上应该落在 object mask 内。

即使这个点被另一块场景几何遮挡，因为 mask 是**所有场景对象的 union mask（联合掩码）**，该像素仍应是 foreground。官方源码明确把所有参与场景 mask 的 mesh 替换成白色发光材质并渲染联合 mask。([GitHub][1])

所以我们直接：

[
\text{mesh surface samples}
\rightarrow
\text{COLMAP direct projection}
\rightarrow
\text{sample masks}
]

根本不需要 triangle rasterizer。

如果 surface samples 有 95% 落在 mask，但 silhouette IoU 只有 0.257：

> **你的 mesh rasterizer 有 bug。**

如果某些 OBJ object block 95% 在 mask，而另外一些只有 5%：

> **mesh export scope 与 mask scope 不一致。**

如果所有 block 都大量投到 mask 外：

> **camera mapping/projection chain 有问题。**

这次可以一刀切开。
