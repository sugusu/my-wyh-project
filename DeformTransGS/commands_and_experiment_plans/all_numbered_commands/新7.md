## 一、为什么现在不能接受 J4 FAIL

三个 case：

```text
K0 optimizer steps = 500
K1 optimizer steps = 500
K2 optimizer steps = 500
```

全部刚好 500。

你的冻结规则明明是：

> 每 100 step 做一次完整 16 TRAIN 视角评估；连续 5 次评估没有出现更优结果，才 early stop（提前停止）。

但是：

```text
K0 selected checkpoint = 500
K1 selected checkpoint = 500
K2 selected checkpoint = 500
```

而且：

[
2.01913145\rightarrow2.01889868
]

[
2.08732496\rightarrow2.08716366
]

也就是说 **500 step 仍然是当前最佳 checkpoint**。

逻辑上：

```text
step500 = new best
→ patience counter 应重置
→ 不可能在 step500 因 500-step patience 停止
```

除非代码写错。

更扎眼的是：

```text
P0 step500 = NOT_SELECTED/FIXED_FOOTPRINT
P1 step500 = NOT_SELECTED/FIXED_FOOTPRINT
```

协议要求实际跑 P0/P1 两个 500-step diagnostic（诊断实验），输出 PSNR 和 tau_eq，然后决定是否固定 footprint（Gaussian 覆盖形状）。

现在没有数字，却直接：

```text
footprint policy = FIXED_FOOTPRINT
R4 = PASS
```

这也不正常。

第三个异常：

```text
_occupancy      LR = 1e-6
_opacity        LR = 1e-6
_transmissivity LR = 1e-6
_features_dc    LR = 1e-6
```

四个语义完全不同的参数组，学习率一模一样，全是 (10^{-6})。

不一定绝对错误。

但考虑到 500 step 后 loss 只下降：

[
\frac{2.019131-2.018899}{2.019131}
\approx0.0115%
]

这必须重新追踪 RT-Splatting 源码里的**真实参数组学习率映射**。

所以现在正式状态应该改为：


```python
patience = 500
```

```python
if iteration >= 500:
```
