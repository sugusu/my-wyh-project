## 为什么我觉得这是 camera-space normal？

因为在针孔相机坐标中：

[
z
]

就是相机朝前方向。

你通过 depth backprojection（深度反投影）得到：

[
P_c(u,v)
========

zK^{-1}
\begin{bmatrix}
u\v\1
\end{bmatrix}
]

然后：

[
d_u=P_c(u+1,v)-P_c(u-1,v)
]

[
d_v=P_c(u,v+1)-P_c(u,v-1)
]

[
n_c=d_u\times d_v
]

如果局部 depth 很平滑，大量 normal 自然会接近：

[
(0,0,\pm1)
]

**这是相机坐标系。**

每个 camera 都会得到自己的：

[
\pm z_{\rm camera}
]

但如果你直接把这些数组拼起来，当成 world normal：

[
n_{\rm world}=n_c
]

那么所有相机都会贡献：

[
(0,0,\pm1)
]

于是刚好得到你现在的：

[
\boxed{
global\ scatter\ rank\approx1
}
]

和：

[
\boxed{
pairwise\ angle\approx0
}
]

这个症状简直太典型了。

# 三、官方 TSGS 相机代码进一步加强了这个怀疑

TSGS 官方相机代码明确构造的是 world-to-view transform（世界到相机变换），`getWorld2View2` 的旋转块使用 `R.transpose()`，随后 Camera 将矩阵转置以适配 CUDA 的 row-vector（行向量）约定；`camera_center` 则来自 `world_view_transform.inverse()`。([GitHub][1])

也就是说：

> camera space 和 world space 在官方管线里是明确区分的。

如果 depth normal 是从：

[
P_c=zK^{-1}p
]

计算出来的，那么它必须经过 camera-to-world rotation（相机到世界旋转）才能参与跨视角融合。

在 TSGS 当前 row-vector（行向量）约定下，最稳妥的做法不是猜 `cam.R` 要不要转置，而是直接使用：

[
V^{-1}
======

\texttt{inverse(world_view_transform)}
]

取纯旋转块。

对于 row normal：

[
\boxed{
n_w
===

n_c
V^{-1}_{3\times3}
}
]

然后重新归一化。

如果代码使用 column vector（列向量），等价写法应按对应转置处理。

**不要再凭记忆写 `R @ n` 或 `R.T @ n`。直接从实际 `world_view_transform` 做 roundtrip unit test（往返单元测试）。**

官方 TSGS 的 Gaussian `get_normal` 明确将 Gaussian normal 标为 world coordinate（世界坐标）；它从最小 scale 轴取得方向后，直接与 `camera_center - xyz` 这种世界坐标向量做点积。([GitHub][2])

所以：

> **你要把 depth normal 和 TSGS Gaussian normal 比较，它们必须先处在同一个 world coordinate。**
