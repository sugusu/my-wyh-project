# 一、修正 Stage 2B-X 根因

当前 root cause report 的大方向正确：

```text
recovery pipeline shared training logic is incorrect
```

但具体根因：

```text
missing gaussians.update_learning_rate(iteration)
```

不准确。

根据 TSGS 官方训练代码，正式训练循环中的：

```python
# gaussians.update_learning_rate(iteration)
```

本身已被注释。

真正执行的是：

```python
gaussians.selective_learning_rate_control(
    iteration,
    15000,
    nofix_position=opt.nofix_position,
    nofix_opacity=opt.nofix_opacity,
    nofix_param=opt.nofix_param,
    nofix_scaling=opt.nofix_scaling,
    nofix_rotation=opt.nofix_rotation,
)
```

因此正式根因修改为：

```text
RECOVERY_STAGE2_TRAINING_POLICY_MISMATCH
```

具体含义：

> custom recovery training loop 没有复现 TSGS iteration > 15000 时的 selective learning-rate/freezing policy，导致本应冻结的 geometry/opacity parameter groups 继续更新。

禁止仅执行：

```python

```

然后直接重跑500步。

# 七、修复 Root Cause Report

修改：

```text
src/prune/build_recovery_collapse_report.py
```

当前错误结论：

```text
Learning rate wrong because update_learning_rate is missing.
```

替换为：

```text
The custom recovery loop did not reproduce TSGS's
post-15k selective learning-rate control.

Official TSGS invokes selective_learning_rate_control()
at each training iteration. At iteration > 15000 this policy
updates the xyz schedule and selectively freezes parameter groups
according to the original nofix_* configuration.

The custom recovery loop skipped this policy and left parameter
groups trainable with restored non-zero learning rates.

This caused schedule_control and all pruning methods to undergo
the same catastrophic parameter drift.
```

新的 root cause enum：

```text
RECOVERY_STAGE2_TRAINING_POLICY_MISMATCH
```

保留：

```text
checkpoint schema consistent
optimizer xyz LR restored correctly
Gaussian count stable
pruning is not the root cause
```

删除未经验证的：

```text
f_dc LR is ~100x too high
opacity LR should decay
```

除非实际本地代码和 LR trace 能直接证明。
