# 二、`cfg_args MISMATCH` 大部分是假警报

你的审计报告说：

```text
iterations        missing
use_asg           missing
delight_iterations missing
nofix_position    missing
...
```

但我查了 TSGS 官方源码。

问题出在 TSGS 自己保存 `cfg_args` 的方式。

`training()` 接收到的是：

```python
training(
    lp.extract(args),
    op.extract(args),
    pp.extract(args),
    ...
)
```

然后：

```python
tb_writer, logger = prepare_output_and_logger(dataset)
```

这里传进去的 `dataset` 实际只是：

```text
ModelParams
```

最后 `cfg_args` 保存的也是这个 `dataset`。因此 `cfg_args` 主要有 `sh_degree`、`resolution`、`eval`、`delight`、`normal` 等 ModelParams 字段；而 `iterations`、`use_asg`、`delight_iterations`、`nofix_*` 属于 OptimizationParams，本来就不会完整出现在这个 `cfg_args` 中。 ([GitHub][1])

所以现在这个结论：

```text
OFFICIAL_CFG_MISMATCH
```

**不能成立。**

比如：

```text
use_asg=<missing>
```

不能解释为：

```text
use_asg=False
```

正确解释应该是：

```text
cfg_args cannot verify use_asg
```

TSGS 官方训练代码还会把完整命令行写进训练日志：

```python
command = " ".join(sys.argv)
logger.info(f"Command: {command}")
```

所以正式配置审计应该以 **training log 中的 `Command:` 为主证据**，而不是只看 `cfg_args`。 ([GitHub][1])

这点一定要修。

## 我的客观判断

现在最关键的新发现不是“OOM”本身，而是：

> **我们第二次训练虽然补了 `delight/normal/use_asg`，但依然没有精确复刻官方 `scripts/run_translab.py` 实际构造的训练命令。**

官方 wrapper 默认还会传 `--mask_background` 和 `--use_transparencies_map`。 ([GitHub][2])

而你的 Gaussian 数量：

```text
551,051 @ 7k
→
2,084,764 @ 10k
```

已经明显进入“高斯数量爆炸 + OOM”的状态；官方训练直到 15k 前仍会继续 densification。 ([GitHub][1])

所以我建议非常明确：

**先别改 densification 参数，也别用 7k checkpoint 续训。修正审计脚本，复原 exact wrapper 命令，从 0 重跑 scene_01。**

这一次如果 exact 配置还在 V100 32GB 上 OOM，我们再正大光明地进入“资源适配 baseline”；否则又会陷入前面那种“以为在跑官方配置，其实少了两个 flag”的循环。
